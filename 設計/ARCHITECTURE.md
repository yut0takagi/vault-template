# Vault Architecture — 設計思想書

> **このフォルダを見れば、全部わかる。**
> 10年後の自分が読んでも、何がどこにあり、なぜそうなっているかが分かること。

---

## 第1章 思想

### 1.1 Vaultとは何か

Vaultは「第二の脳」ではない。**自分の仕事のOS**である。

OSには3つの責務がある：
1. **記憶** — 過去の決定・議論・学びを失わない
2. **実行** — 繰り返す作業を自動で回す
3. **発見** — 必要な情報に最短で辿り着ける

この3つが揃って初めて「このフォルダを見れば全部わかる」が成立する。

### 1.2 設計原則

| # | 原則 | 意味 | 違反例 |
|---|------|------|--------|
| 1 | **テキストは中に、バイナリは外に** | mdファイルはVault内に実体を置く。音声・画像・スライドは外部ストレージに置き、Vaultからリンクで参照する | 130MBのm4aをgit管理する |
| 2 | **全部入れるのではなく、全部辿れる** | 外部サービスの情報はVault内のmdからリンクで到達できればよい。コピーを作らない | Slack投稿をmdに全文コピー |
| 3 | **フォルダは役割、タグは性質** | フォルダは「これは何か」（議事録/ナレッジ/タスク）。タグは「何について」 | フォルダでトピック分類する |
| 4 | **1ファイル1目的** | 1つのmdは1つの議事録、1つのナレッジ、1つのタスクを表す。複合しない | 議事録の中にタスク管理テーブルを書く |
| 5 | **自動化は発見可能であること** | どの自動化がどこにあるか、1ファイルで全容を把握できる | GASのURLをスキル内にだけ書く |
| 6 | **フォーマットは最小公約数** | markdownとプレーンテキスト。特定ツールが消えても読める | 専用プラグインでしか開けないデータ |
| 7 | **命名で中身が分かる** | ファイル名だけで日付・種類・内容が推測できる | `メモ.md`, `test2.md` |

### 1.3 10年後も壊れないために

- **ツールは変わる。フォーマットは残る。** ObsidianもClaude Codeも10年後は存在しないかもしれない。だからデータは常にmarkdown（プレーンテキスト）で持つ。
- **自動化は補助輪。** 自動化が壊れてもVaultは使える。手動でファイルを作成して同じ構造に置けば、すべて動く。
- **構造は足し算で進化させる。** フォルダやタグの大規模リネームはしない。新しい概念は新しいフォルダ/タグとして追加する。

## 第2章 フォルダ構成

### 2.1 構成図

```
vault/
├── CLAUDE.md                     ← AI向け全体指示
├── 設計/                         ← 設計・設定・テンプレート
│   ├── ARCHITECTURE.md           ← 本ファイル（設計思想書）
│   ├── タグ定義.md               ← タグ運用ルール
│   └── テンプレート/             ← Templater用テンプレート
│
├── 議事録/                       ← 会議・面談の議事録
├── ナレッジ/                     ← 知識ベース
│   ├── 技術/
│   ├── ツール/
│   └── リサーチ/
├── inbox/                        ← 未整理素材
│   ├── raw/                      ← 生の取り込みデータ
│   ├── temp/                     ← 一時ファイル
│   ├── まとめ済み/               ← 処理済みインプット
│   └── ストック/                 ← 保留・あとで読む
│
├── タスク/                       ← タスク管理
│   └── 詳細/                     ← 各タスクの個票
├── 日報/                         ← 日報・振り返り
├── ログ/チャット/                ← AIチャットログ（月次）
├── 制作/スライド/                ← 発表資料
├── 添付/                         ← 画像等のバイナリ
├── アーカイブ/                   ← 処理済みファイル退避先
├── Excalidraw/                   ← 図・ホワイトボード
├── Clippings/                    ← Webクリップ
│
├── .claude/                      ← Claude Code 自動化インフラ
│   ├── skills/
│   ├── commands/
│   └── agents/
│
└── .obsidian/                    ← Obsidian設定
```

### 2.2 フォルダの役割定義

| フォルダ | 何を置くか | 何を置かないか | ライフサイクル |
|----------|-----------|---------------|---------------|
| `議事録/` | 会議・面談の議事録md | 音声ファイル、下書き | 永久保持 |
| `ナレッジ/` | 再利用可能な知見 | 一次情報（議事録そのもの） | 定期更新 |
| `inbox/` | 未整理の素材 | 完成したナレッジ | temp/は7日、まとめ済みは30日 |
| `タスク/詳細/` | 各タスクのDoD・進捗 | 完了後1ヶ月経過したタスク | 完了→アーカイブ |
| `日報/` | 日報・週次/月次レビュー | 議事録、ナレッジ | 永久保持 |
| `制作/スライド/` | .pptx + 素材md | 大量の画像素材 | 永久保持 |
| `添付/` | md内から参照される画像・音声 | 処理済み音声 | 参照なくなったら削除 |
| `設計/` | 設計書・テンプレート | ナレッジ、タスク | 改訂で進化 |
| `アーカイブ/` | 処理済みファイル | アクティブなファイル | 長期保管 |

### 2.3 新しいフォルダを作るとき

以下のすべてを満たす場合のみ、新フォルダを作る：
1. 既存フォルダのどれにも属さない**役割**がある
2. 10個以上のファイルが見込める
3. 本ファイルの2.2に行を追加する

満たさない場合はタグで分類する。

## 第3章 ファイル命名規則

| 種類 | パターン | 例 |
|------|---------|-----|
| 議事録 | `YYYY-MM-DD_{タイトル}.md` | `2026-03-31_プロジェクト定例.md` |
| 日報 | `YYYY-MM-DD.md` | `2026-03-31.md` |
| 週次レビュー | `週次レビュー-YYYY-WXX.md` | `週次レビュー-2026-W14.md` |
| 月次レビュー | `月次レビュー-YYYY-MM.md` | `月次レビュー-2026-03.md` |
| タスク詳細 | `{動詞} {対象}.md` | `構築 音声アーカイブ.md` |
| ナレッジ | `{トピック名}.md` | `Claude Code 総合ガイド.md` |
| ナレッジ（外部） | `YYYY-MM-DD-{トピック名}.md` | `2026-04-02-AI駆動開発.md` |
| チャットログ | `YYYY-MM.md` | `2026-03.md` |
| スライド | `{タイトル}.pptx` | `知識管理革命.pptx` |

**禁止:** `メモ.md`, `test.md`, `無題.md`（内容が推測できない名前）、連番（`メモ1.md`）

## 第4章 メタデータ設計

### 4.1 frontmatter 統一ルール

すべてのmdファイルは YAML frontmatter を持つ。

```yaml
---
tags: [構造タグ, カテゴリタグ, 内容タグ1, 内容タグ2]
---
```

- tags は `[タグ1, タグ2]` 形式。`"#タグ"` とハッシュを付けない
- 日付は `YYYY-MM-DD` 形式

### 4.2 種類別 frontmatter

#### 議事録
```yaml
---
tags: [議事録, {会議種別}, {トピック}]
related: ["[[関連ナレッジ名]]"]
---
```

#### ナレッジ
```yaml
---
tags: [ナレッジ, {カテゴリタグ}, {内容タグ1}, {内容タグ2}]
collected: YYYY-MM-DD
source_url: "https://..."
---
```

#### タスク詳細
```yaml
---
PJT: {PJT名}
期限: YYYY-MM-DD
優先度: Urgent / High / Medium / Low
status: 未着手 / 調査中 / 設計中 / 作業中 / レビュー待ち / 完了
---
```

## 第5章 データ管理

### 5.1 バイナリファイルの扱い

| ファイル種別 | Git管理 | ライフサイクル |
|-------------|---------|--------------|
| `.md` | する | 永久 |
| `.m4a`/音声 | **しない** | 処理後→外部退避→ローカル削除 |
| `.pptx` | する（サイズ注意） | 永久（10MB超は外部退避検討） |
| `.png/.jpg` | する（サイズ注意） | 参照なくなったら削除 |

### 5.2 アーカイブ・削除ポリシー

| 対象 | 条件 | アクション |
|------|------|-----------|
| `inbox/temp/` | 7日経過 | 削除 |
| `inbox/まとめ済み/` | 30日経過 | アーカイブ or 削除 |
| `タスク/詳細/` 完了タスク | 完了後1ヶ月 | アーカイブに移動 |
| `ログ/チャット/` | — | 永久保持 |
